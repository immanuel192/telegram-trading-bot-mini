# Implement Telegram Client using mtcute

## Summary
Implement the core Telegram client functionality in `telegram-service` using the `mtcute` library. This will enable the service to connect to Telegram via MTProto, listen for new messages from specific channels, persist them to MongoDB, and publish them to Redis Streams for downstream processing.

## Motivation
`mtcute` is a modern, pure TypeScript/JavaScript MTProto client that is lightweight, type-safe, and actively maintained. It offers a better developer experience and performance compared to older libraries like `telegram` (GramJS) or `telegraf` (Bot API only). Using MTProto allows us to act as a userbot if needed, which is often required for reading from channels without being an admin.

## Scope
- Install and configure `mtcute` and `@upstash/redis`.
- Implement a `TelegramClientService` wrapper service.
- Handle authentication and session management.
- Implement message listening, processing, and persistence.
- Implement Redis Stream abstraction layer in `@libs/shared/utils`:
  - `IStreamPublisher` and `IStreamConsumer` interfaces
  - `RedisStreamPublisher` and `RedisStreamConsumer` implementations
  - `StreamMessage` type with versioning
  - `MessageType` and `StreamTopic` enums
- Publish processed messages to Redis Streams (`stream:messages`).
- Add Redis container to `docker-compose.yml` for local development.
- Update configuration to support both local Redis and Upstash Redis (production).
- Add comprehensive unit tests for stream publishing.

## Implementation Details

### Redis Stream Architecture

#### Stream Design
- **Stream Topic**: `messages` - single stream for all Telegram messages
- **Message Format**:
  ```typescript
  {
    version: "1.0",
    type: MessageType.NEW_MESSAGE,
    payload: TelegramMessage // Full message object with all fields
  }
  ```
- **Entry ID**: Auto-generated by Redis (timestamp-based like `1526919030474-0`)
  - Guarantees monotonically increasing IDs for perfect time-ordered processing
  - Built-in sequence numbers handle concurrent writes to the same millisecond
  - No collision detection needed, optimized by Redis
  - Easy range queries for time-based message retrieval

#### Environment Support
- **Local Development**: 
  - Uses Redis 7 Alpine container in `docker-compose.yml`
  - Connection: `http://localhost:6379`
  - No authentication required (empty token)
  - Persistent storage via Docker volume
  
- **Production**: 
  - Uses Upstash Redis (serverless, HTTP-based)
  - Connection: Upstash Redis URL
  - Token-based authentication
  - Same API, seamless transition

#### Abstraction Layer (`@libs/shared/utils/src/stream/`)

**Interfaces** (`interfaces.ts`):
```typescript
// Enums
enum MessageType { NEW_MESSAGE = 'NEW_MESSAGE' }
enum StreamTopic { MESSAGES = 'stream:messages' }

// Core types
interface StreamMessage<T> {
  version: string;
  type: MessageType;
  payload: T;
}

interface RedisStreamConfig {
  url: string;
  token?: string;
}

// Publisher interface
interface IStreamPublisher {
  publish<T>(topic: StreamTopic, message: StreamMessage<T>): Promise<string>;
  close(): Promise<void>;
}

// Consumer interface
interface IStreamConsumer {
  consume<T>(
    topic: StreamTopic,
    groupName: string,
    consumerName: string,
    handler: (message: StreamMessage<T>, id: string) => Promise<void>
  ): Promise<void>;
  stop(): Promise<void>;
  close(): Promise<void>;
}
```

**Publisher Implementation** (`redis-stream-publisher.ts`):
- Uses `@upstash/redis` client (works with both local Redis and Upstash)
- Serializes message to Redis Stream format
- Uses `XADD` with auto-generated ID (`*`)
- Returns stream entry ID for tracking
- No-op `close()` for HTTP-based client compatibility

**Consumer Implementation** (`redis-stream-consumer.ts`):
- Creates consumer groups with `XGROUP CREATE`
- Polls messages using `XREADGROUP`
- Processes messages in batches (count: 10, block: 5000ms)
- Acknowledges processed messages with `XACK`
- Handles errors gracefully with retry logic
- Supports graceful shutdown

**Service Integration** (`apps/telegram-service/src/services/telegram-client.service.ts`):
```typescript
export class TelegramClientService implements ITelegramClientService {
  constructor(
    private readonly configRepository: ConfigRepository,
    private readonly telegramChannelRepository: TelegramChannelRepository,
    private readonly telegramMessageRepository: TelegramMessageRepository,
    private readonly streamPublisher: IStreamPublisher, // ← Interface-based
    private readonly logger: LoggerInstance
  ) {}

  private async processMessage(task: MessageTask): Promise<void> {
    try {
      // ... extract and persist message

      // Publish to Redis Stream
      await this.streamPublisher.publish(
        StreamTopic.MESSAGES,
        {
          version: '1.0',
          type: MessageType.NEW_MESSAGE,
          payload: telegramMessage,
        }
      );

      this.logger.debug({ channelCode, messageId }, 'Message published to stream');
    } catch (error) {
      this.logger.error({ err: error }, 'Error processing message');
      throw error; // Re-throw to trigger queue error handler
    }
  }
}
```

- **Decoupled Architecture**: Telegram ingestion separated from message processing
- **Horizontal Scaling**: Multiple consumers can process messages in parallel
- **Consumer Groups**: Redis Streams ensure each message processed exactly once
- **Backpressure Handling**: Queue-based processing prevents overwhelming downstream services

### Reliability
- **Persistence**: Redis Streams persist messages until acknowledged
- **Replay Capability**: Can reprocess messages from any point in time
- **Error Recovery**: Failed messages can be retried without data loss
- **Monitoring**: Stream length and consumer lag provide operational visibility

- **Version Support**: Message versioning allows schema evolution

## Recent Updates

### Message Payload Schema (2025-11-21)

**Changed Message Format**: The payload structure has been updated to be more lightweight and focused:

```typescript
// OLD format (full TelegramMessage object)
{
  version: "1.0",
  type: MessageType.NEW_MESSAGE,
  payload: TelegramMessage // Entire message object
}

// NEW format (lightweight reference with expiry)
{
  version: "1.0",
  type: MessageType.NEW_MESSAGE,
  payload: {
    messageId: number,      // Reference to message in MongoDB
    exp: number            // Expiry timestamp (24 hours)
  }
}
```

**Rationale**:
- **Reduced Stream Size**: Only essential data in stream, full message in MongoDB
- **Single Source of Truth**: MongoDB is the authoritative source for message data
- **Expiry Handling**: Consumers can skip expired messages (24-hour TTL)
- **Decoupling**: Consumers fetch full message from DB using `messageId`

### Retry Logic with Exponential Backoff

**Retry Configuration**:
```typescript
interface RetryConfig {
  maxRetries: number;          // Default: 3
  initialDelayMs: number;      // Default: 1000ms
  maxDelayMs: number;          // Default: 30000ms
  backoffMultiplier: number;   // Default: 2 (exponential)
}
```

**Retry Flow**:
1. **First Attempt**: Process message immediately
2. **Retry 1**: Wait 1s, retry
3. **Retry 2**: Wait 2s (1s × 2), retry
4. **Retry 3**: Wait 4s (2s × 2), retry
5. **Max Retries Exceeded**: 
   - Log error with context
   - Capture exception via `IErrorCapture`
   - Acknowledge message (prevent infinite loop)
   - TODO: Move to Dead Letter Queue

**Error Handling**:
- **Validation Failures**: Acknowledged immediately (invalid data)
- **Expired Messages**: Acknowledged immediately (too old)
- **Processing Errors**: Retried with backoff
- **Max Retries**: Acknowledged with error capture (needs DLQ)

### Consumer Message Processing Flow

```
┌─────────────────────────────────────────────────────────────┐
│ 1. XREADGROUP (poll for messages)                          │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. Parse message fields (version, type, payload)           │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│ 3. Validate message (IMessageValidator.validate)           │
│    ├─ Invalid? → Log + Capture Error + ACK + Skip          │
│    └─ Valid? → Continue                                     │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│ 4. Check expiry (IMessageValidator.isExpired)              │
│    ├─ Expired? → Log + Capture Error + ACK + Skip          │
│    └─ Not Expired? → Continue                               │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│ 5. Process with retry (exponential backoff)                │
│    ├─ Success? → ACK + Done                                 │
│    ├─ Error + Retries Left? → Wait + Retry                  │
│    └─ Max Retries? → Log + Capture + ACK (TODO: DLQ)        │
└─────────────────────────────────────────────────────────────┘
```
